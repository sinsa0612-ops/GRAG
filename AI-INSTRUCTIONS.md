# 🤖 AI 에이전트 개발 및 소통 지침서 (AI-INSTRUCTIONS.md)

이 문서는 이 프로젝트의 '개발 및 코딩'을 담당하는 AI(안티그래비티 등 CLI 에이전트)가 반드시 준수해야 할 헌법이자 아키텍처 설계도입니다. 작업을 시작하기 전 항상 이 문서를 최우선으로 숙지하라.

## 📌 0. 핵심 철학 (Core Philosophy)
- **방향성(Architecture):** 코드는 철저히 책임과 역할 단위로 격리된 **'1모듈 1R&R'** 구조를 유지한다.
- **검증(Validation):** 모든 변경 사항은 제로 컨텍스트 상태에서 '스모크 테스트'를 거쳐야 한다.
- **연속성(Continuity):** 세션이 종료되거나 전환될 때는 인간-AI 간의 주도권을 분리하여 문서를 남긴다.
- **단순함(Simplicity):** 의심스러울 때는 항상 더 단순한 쪽을 택한다. 복잡함은 비용이다.

## ⚖️ 0-1. 규칙 적용 우선순위 및 충돌 해소 (Conflict Resolution)
이 헌법은 너를 옭아매기 위한 것이 아니라 방향을 잡아주기 위한 것이다. 규칙들이 서로 당길 때는 아래 원칙으로 판단하라.
- **경계 vs 내부:** 2장·8장의 구조 강제 규칙(어댑터, DAL, 에이전트 간 Pydantic 검증)은 **시스템 경계**(외부 라이브러리 연동, DB 접근, 에이전트 간 데이터 교환)에 적용한다. 그 경계 **안쪽의 내부 로직**은 9장의 KISS 원칙을 우선하여 단순하게 작성한다. 사소한 기능에 풀 아키텍처를 두르지 마라.
- **막혔을 때의 관용 밸브:** 규칙끼리 명백히 충돌하거나, 이 헌법을 그대로 따르는 것이 특정 상황에서 분명히 비합리적이라고 판단되면, **임의로 규칙을 어기지 말고** 충돌 지점과 너의 판단 근거를 사용자에게 보고하고 결정을 구하라. (9장 '질문 우선'의 연장이다.)

## 🚫 1. 절대 금지 사항 (Strict Prohibitions)
- **Git 통제:** 너는 `git add`, `git commit`, `git push` 등 버전 관리 명령어를 절대 직접 실행하지 않는다. 작업이 끝나면 사용자에게 보고하고, 사용자가 직접 커밋하도록 대기하라.
- **하드코딩 금지:** 프롬프트 문자열, API 키, 파일 경로, 환경 설정값을 파이썬 로직 내부에 절대 직접 박지 마라. 모든 설정값은 **설정 계층(Config Layer)과 .env의 단일 출처**에서만 가져온다. (상세: 2-1 설정 중앙화 규칙)
- **프로젝트 루트 하드코딩 금지:** `D:/AI/AIPA/` 같은 절대 경로를 코드에 직접 쓰지 마라. 루트는 항상 `settings.project_root`(런타임에 `__file__` 기준으로 산출)를 참조한다.

## 🧩 2. 아키텍처 및 데이터베이스 규칙
- **1모듈 1R&R (Role & Responsibility):** 무조건적인 함수 단위의 파편화(`1모듈 1기능`)를 지양한다. 에이전트가 전체 맥락을 잃지 않도록, 명확한 도메인과 하나의 책임(역할)을 기준으로 모듈을 설계하라.
- **데이터 접근 계층 (DAL) 격리:** 모든 데이터베이스 접근은 오직 `db_manager.py`를 통해서만 이루어지는 명확한 DAO/Repository Layer 계층 분리 구조를 준수한다. 다른 서브 에이전트 파일에 raw SQL 쿼리를 직접 작성하여 계층을 오염시키지 마라.
- **입구 검증 (Schema Validation):** DB에 데이터를 넣기 전이나 에이전트 간 데이터를 주고받을 때는 반드시 `Pydantic`을 이용해 데이터 규격을 엄격하게 검증하라.
  - **[경계 - 필수 적용]:** 외부 API 요청 데이터, DB(데이터베이스) 최종 저장 직전, LLM 구조화된 출력(Structured Output)을 최초로 파싱 및 검증할 때는 반드시 Pydantic 스키마를 통해 엄격히 검증한다.
  - **[내부 - 적용 면제]:** 이미 입구에서 검증이 완료된 데이터가 내부 서브 에이전트나 비즈니스 로직 함수 간에 패싱될 때는 무거운 Pydantic 모델을 강제하지 않는다. 이 구간에서는 파이썬 기본 딕셔너리(`dict`), `dataclass`, 또는 `TypedDict`를 활용하여 가볍고 유연하게 데이터를 전달함으로써, 에이전트의 불필요한 클래스 양산과 코드 비대화를 막는다.

### ⚙️ 2-1. [필수 인프라] 설정 중앙화 (Centralized Config)
- **단일 출처 원칙:** 모든 경로, API 키, 임계값 등 환경 의존적인 값은 설정 계층(Config Layer)의 단일 출처에서 관리한다. 프로젝트 규모에 따라 설정은 여러 파일로 분리할 수 있으나, 환경 의존 값은 반드시 설정 계층을 거치게 한다. 비밀값은 코드가 아니라 `.env` 파일에 두고, `.env`는 `.gitignore`에 포함시킨다.
- **동적 경로 산출:** 프로젝트 루트는 절대 경로를 박지 말고 아래처럼 동적으로 계산하라. (아래는 가장 단순한 단일 파일 예시이다.)

```python
# Example: 단일 Settings 파일 구성 예시
from pathlib import Path
from pydantic_settings import BaseSettings, SettingsConfigDict

class Settings(BaseSettings):
    model_config = SettingsConfigDict(env_file=".env", env_file_encoding="utf-8")
    project_root: Path = Path(__file__).resolve().parent
    temp_dir: Path = Path(__file__).resolve().parent / "temp"
    # api_key: str  # .env 에서 주입

settings = Settings()
```

### 🚨 [핵심 아키텍처 규칙] agy CLI 통신 패턴 (File-based IPC)
- **절대 금지:** `agy` CLI 서브 프로세스 호출 시, 비대화형 환경의 버그(0바이트 출력)를 피하기 위해 결과값을 `stdout`으로 수집하려고 시도하지 마십시오.
- **표준 패턴 (File IPC 강제):** 안티그래비티의 도구 권한(`write_file`)을 반드시 활용하십시오. LLM에 전달하는 자연어 프롬프트 마지막에 **"결과를 다른 설명 없이 (런타임에 주입된 임시파일 경로) 파일에만 순수 JSON으로 작성하고 저장하라"**고 명시적으로 지시해야 합니다.
- **고유 임시파일 강제 (경합 방지):** 출력 파일명을 `temp_output.json`처럼 고정하지 마라. 매 호출마다 `tempfile`로 **고유한 임시파일**을 생성하여 동시 작업 간 덮어쓰기(race condition)를 원천 차단한다.
- **파이썬 오케스트레이션:** 서브 프로세스가 정상 종료(Exit Code 0)되면 생성된 임시파일을 읽어 JSON을 파싱하고, **성공·실패 여부와 무관하게 `try/finally`로 반드시 파기**하는 구조를 엄격하게 준수해야 합니다.

```python
# agy 호출 표준 오케스트레이션
import json, os, tempfile, subprocess
from config import settings

def call_agy(prompt: str) -> dict:
    settings.temp_dir.mkdir(parents=True, exist_ok=True)
    fd, tmp_path = tempfile.mkstemp(suffix=".json", dir=settings.temp_dir)
    os.close(fd)  # 고유 파일명 → 동시 호출에도 충돌 없음
    try:
        full_prompt = (
            f"{prompt}\n\n결과를 다른 설명 없이 아래 경로 파일에만 "
            f"순수 JSON으로 작성하고 저장하라: {tmp_path}"
        )
        subprocess.run([...], check=True)  # Exit Code 0 보장 (check=True)
        with open(tmp_path, encoding="utf-8") as f:
            return json.load(f)
    finally:
        if os.path.exists(tmp_path):
            os.remove(tmp_path)  # 성공/실패 무관 무조건 파기
```

## 🛠️ 3. 기술 규격 (Technical Specifications)
- **런타임 환경:** 반드시 **Python 3.12**의 최신 표준 문법과 타입 힌팅(Type Hinting)을 준수하라.
- **네이밍 컨벤션 (PEP 8 엄격 준수):**
  - 변수명, 함수명, 파일명: 반드시 `snake_case` (소문자와 언더바)만 사용한다.
  - 클래스명: 반드시 `PascalCase` (단어 첫 글자 대문자)를 사용한다.
  - **내 코드 내부에서는 `camelCase`를 어떠한 경우에도 혼용하지 마라.**
  - **유일한 예외:** 외부 API의 JSON이 `camelCase`로 들어올 때만, Pydantic의 `alias`/`populate_by_name`으로 받아 **내부에서는 즉시 `snake_case`로 변환**한다. camelCase가 내 코드 본문까지 흘러 들어오게 두지 마라.
- **로깅 규칙:** 진단/추적 출력에 `print()`를 쓰지 마라. 반드시 표준 `logging` 모듈을 사용하고, 로그 레벨(DEBUG/INFO/WARNING/ERROR)을 의미에 맞게 구분하라.
- **의존성 관리:** 패키지는 가상환경(`.venv`) 안에서만 설치하고, 추가/변경 시 `requirements.txt`(또는 `pyproject.toml`)에 명시적으로 고정(pin)하라. 전역 환경을 오염시키지 마라.
- **자동 정렬:** 코드 작성이 완료되면 터미널 포매터(예: `ruff format` 또는 `black`)를 실행하여 형식을 강제로 통일하라.

## 🧠 4. 코드 수정 전 제약 조항

- **충동적 코딩 및 땜빵 금지:** 에러나 버그를 마주했을 때 원인을 파악하지 못한 상태에서 임시 우회 코드나 아키텍처 원칙을 무너뜨리는 수정(야매 코딩)을 하지 마라. 원인을 설명할 수 없다면 수정하지 말고 사용자에게 보고하라.
- **구조 이해 보고서 선행:** 구조, 데이터 흐름, 아키텍처 규칙, 모듈 간 계약(인터페이스), DB 스키마를 변경하는 작업 전에는 반드시 다음 3가지를 분석한 **[구조 사전 이해 보고서]**를 3줄 요약 형태로 사용자에게 먼저 제출하여 확답을 얻어야 한다.
  1. 현재 발생한 버그의 근본 원인 분석
  2. 수정 대상 로직과 연계된 기존 데이터 흐름 및 아키텍처 규칙
  3. 기존 코드를 해치지 않고 정석대로 해결할 구현 방안
- **구현 계획서 작성:** 신규 기능, 파일 3개 이상 수정, 구조 변경, DB 스키마 변경, Agent 인터페이스 변경, 대규모 리팩토링은 구현 전 `implementation_plan.md`를 작성한다. 필요시 상위 추론 모델 또는 Architect Agent 검토를 권고할 수 있다.
- **검증서 작성:** 사용자 기능에 영향을 주는 변경이 완료되면 `walkthrough.md`를 작성하여 변경 사항과 검증 결과를 기록한다.
- **면제 대상:** 오타 수정, 포매팅, 주석·문서 수정, 로그 문구 변경, 승인된 계획의 단순 이행, 구조와 로직에 영향을 주지 않는 수정은 위 절차를 생략할 수 있다. 단, 변경 내역은 `HANDOFF-sub.md`에 기록한다.

## 🧪 5. pytest 기반 검증 워크플로우 (Verification)
- **테스트 러너 단일화:** 이 프로젝트의 기본 테스트 러너는 **`pytest`**이며, 모든 테스트 구동 명령은 `pytest`로 단일화한다.
- **하위 호환성 원칙:** 레거시 프로젝트의 경우, 기존 구축된 `unittest` 기반 테스트 스위트가 존재한다면 이를 수정하지 않고 `pytest`가 그대로 흡수하여 완벽하게 구동(Pass)하도록 호환성을 유지하라. (신규 프로젝트는 처음부터 `pytest` 단일 표준으로 빌드한다.)
- **신규 기능 작성 및 격리 원칙:** 향후 개발되는 모든 신규 기능의 테스트는 아래 2대 원칙을 조합하여 클래스 상속이 없는 **pytest 표준(함수형, Fixture, 내장 assert)**으로 간결하게 작성하라.
  1. **DB 테스트 (인메모리 격리):** DB 테스트 시 실제 파일(`tasks.db`) 오염을 막기 위해, 반드시 `pytest fixture`로 인메모리 DB(`sqlite:///:memory:`)를 동적 주입하여 격리 수행하라.
  2. **독립 검증 (Mocking):** 외부 API나 라이브 LLM 서브프로세스(`agy`) 등 외부 의존성이 있는 로직은 반드시 `mock`으로 차단하여 네트워크 없이 1초 내외로 종료되도록 구현하라.

## 🗣️ 6. 비전공자 맞춤형 소통 규칙 (Communication)
- **사용자/기능 중심 설명 (★가장 중요):** 코드 수정 후 "어떤 파일의 라우터를 고쳤는지" 설명하지 마라. **"사용자 화면이나 결과물에서 무엇이 바뀌었고, 기능적으로 어떤 변화가 체감되는지"**를 최우선으로 설명하라.
- **일상적인 비유 사용:** 개발 전문 용어를 사용할 때는 반드시 일상적인 비유(요리, 건축, 회사 생활 등)를 들어 먼저 설명하라.
- **밀당 원칙 (발산과 수렴):** "아이디어를 내봐"라고 할 때는 규칙을 풀고 기발한 구조를 제안하며, "코드를 작성해"라고 할 때는 위의 모든 규격을 기계처럼 엄격하게 지켜라.

## 💾 7. 이원화된 세션 인수인계 규칙 (Handoff-Main & Handoff-Sub)
- 오늘의 코딩 작업 세션을 마칠 때나 사용자가 "오늘은 여기까지"라고 지시할 경우, 컨텍스트 오염을 막기 위해 인수인계 문서를 명확히 분화하여 관리한다.

### 7-1. `HANDOFF-main.md` (인간 주도권: 마일스톤 관리)
- 이 문서는 오직 **사용자(인간)가 최종 확정하고 승인한 핵심 마일스톤, 검증된 아키처 구조, 승인된 규칙**만 기록된다.
- AI 에이전트는 사용자의 명시적 승인 없이 이 파일의 마일스톤 내용을 임의로 수정하거나 조작할 수 없다.

### 7-2. `HANDOFF-sub.md` (AI 주도권: 실무 변경 로그 관리)
- 이 문서는 **AI 에이전트가 매 턴마다 작업한 세부 변경 이력, 현재 태스크 상태, 발견된 미해결 문제(Known Issues), 다음 세션에 이어서 해야 할 구체적인 실무 작업(Next Steps)**이 기록된다.
- 매 세션 종료 시 이 문서를 최신화하여 맥락의 연속성을 보장하라.

### 🗂️ 7-3. 내부 아티팩트 동기화 규칙 (Artifact Synchronization)
- **가시성 확보:** 너는 내부적으로 생성하는 기획 및 작업 문서(`task.md`, `implementation_plan.md`, `walkthrough.md` 등)를 너의 숨겨진 brain 폴더에만 두지 마라.
- **루트 폴더 복제 (Mirroring):** 해당 아티팩트들이 생성되거나 업데이트될 때마다, 반드시 프로젝트 루트 폴더(`settings.project_root`)에도 동일한 파일명으로 복사본을 작성하여 덮어쓰기(Overwrite) 해라.
- 이를 통해 사용자가 워크스페이스 최상단에서 에이전트의 현재 계획과 작업 내역을 실시간으로 모니터링할 수 있도록 유지해야 한다.

## 🔌 8. 외부 라이브러리 및 오픈소스 연동 (External Dependencies)
- **어댑터 패턴(Adapter Pattern) 의무화:** `MarkItDown` 등 외부 오픈소스나 서드파티 라이브러리를 도입할 때는 절대 시스템 전반에 걸쳐 무분별하게 직접 호출(`import`)하지 마라.
- **격리 원칙:** 반드시 해당 라이브러리를 감싸는 전용 어댑터 모듈(예: `document_parser.py`)을 생성하여 내부에서만 격리 실행되도록 구성하라. 다른 메인/서브 에이전트들은 외부 라이브러리의 존재를 완벽히 모른 채, 어댑터가 반환하는 표준화된 데이터(예: Markdown 문자열, JSON)만 받아 사용해야 한다.

## 🧠 9. 핵심 코딩 행동 강령 (Core Coding Behaviors)
- **질문 우선 (Ask Before Coding):** 요구사항이 모호하거나 구현 방식에 대한 정보가 부족할 때는 섣불리 추측하여 코드를 작성하지 마라. 반드시 먼저 사용자에게 질문하여 의도를 명확히 파악하라.
- **단순함 유지 (KISS Principle):** 오버엔지니어링을 철저히 배제하라. 100줄로 끝낼 수 있는 명확한 로직을 불필요한 추상화나 과도한 패턴을 사용해 1,000줄로 복잡하게 만들지 마라.
- **폭발 반경 최소화 (Minimize Blast Radius):** 특정 버그를 수정하거나 단일 기능을 추가할 때, 요구받은 부분만 정확히 수정하라. 정상적으로 작동하는 주변 코드는 절대 불필요하게 리팩토링하거나 건드리지 마라.
